## 1. 文档概述

### 1.1 文档目的
本文档旨在明确 **Stream Analyzer 智能视频流分析平台** 雏形阶段 (MVP) 的功能需求、非功能需求及业务逻辑。作为产品设计、开发、测试及验收的基准文件，确保各团队对 MVP 版本的核心价值、功能边界及交互流程达成一致理解。

### 1.2 适用范围
本文档适用于 Stream Analyzer MVP 版本的全生命周期管理，涵盖以下模块：
1. **工作台**：全局数据概览与快捷入口。
2. **推流管理**：视频流接入、配置及状态监控。
3. **检测项目**：AI 算法任务配置与 ROI 区域管理。
4. **告警中心**：实时告警推送、历史记录及处置流程。
5. **工单分发管理**：告警驱动的工单自动生成、分配、流转与闭环。
6. **人员权限分配管理**：细粒度角色权限、团队组织架构与人员管理。
7. **系统管理**：基础配置与日志审计。

### 1.3 术语定义

| 术语 | 定义 |
| :--- | :--- |
| **RTSP/RTMP** | 实时流传输协议，用于视频源接入。 |
| **ROI** | Region of Interest，感兴趣区域，用于限定 AI 检测范围。 |
| **WebSocket** | 全双工通信协议，用于前端实时接收告警推送。 |
| **置信度** | AI 模型对检测结果的确信程度 (0-100%)。 |
| **MVP** | Minimum Viable Product, 最小可行性产品。 |

## 2. 产品背景与目标

### 2.1 市场背景
随着安防与工业物联网的发展，视频数据量呈指数级增长。传统"人盯屏幕"的监控模式已无法应对海量视频流，且存在疲劳漏检、响应滞后等问题。市场急需一套能够实时理解视频内容并自动触发业务闭环的智能化平台。

### 2.2 用户痛点
1. **监控盲区与疲劳**：人工无法 24 小时保持高专注度，夜间或复杂场景易漏检。
2. **告警滞后**：传统方案多为事后录像检索，缺乏事中实时干预能力。
3. **接入复杂**：多协议（RTSP/RTMP/FLV）混合场景下，流媒体管理分散，缺乏统一控制台。
4. **误报干扰**：缺乏灵活的阈值与区域配置，导致树叶晃动、光影变化引发大量无效告警。

### 2.3 产品目标
1. **实时性**：实现从视频帧采集到告警推送的全链路延迟 < 3 秒。
2. **智能化**：MVP 阶段支持 4 种核心视觉算法，准确率 > 90%。
3. **易用性**：5 分钟内完成单路视频流接入与检测任务配置。
4. **稳定性**：支持单节点 50 路 1080P 并发分析，服务可用性 > 99.5%。

### 2.4 核心价值主张
**"看见即感知，感知即响应"**。通过 AI 视觉引擎与 WebSocket 实时通信，将非结构化视频流转化为结构化告警事件，赋能业务实时决策。

## 3. 用户角色与场景

### 3.1 用户角色定义

| 角色 | 职责描述 | 核心关注点 |
| :--- | :--- | :--- |
| **系统管理员** | 负责平台基础配置、组织架构与角色权限分配、工单策略配置、系统健康监控。 | 系统稳定性、权限安全、日志审计、SLA 达标率。 |
| **运维主管** | 负责视频流接入、设备状态维护、检测算法参数调优、工单分配与验收、班组排班管理。 | 在线率、推流稳定性、算法误报率、工单闭环率、排班合理性。 |
| **值班工程师** | 负责告警与工单处置、现场反馈、交接班日志填写。 | 工单处理时效、处置质量、交接完整性。 |
| **业务分析师** | 负责告警与工单数据挖掘、运营报表生成、知识库维护。 | 事件趋势分析、工单统计报表、知识库覆盖率。 |

### 3.2 典型使用场景

*场景 A：运维工程师 - 新园区监控接入*
1. **背景**：新园区部署了 20 个 IPC 摄像头，需接入 Stream Analyzer 进行周界防护。
2. **操作**：工程师在"推流管理"中批量导入 RTSP 地址，设置统一分辨率 1080P/25FPS。
3. **结果**：系统自动探测流状态，20 路视频在 3 分钟内全部上线，并在"检测项目"中一键应用"周界入侵"模板。

*场景 B：业务分析师 - 实时告警处置*
1. **背景**：深夜值班，需第一时间响应仓库异常。
2. **操作**：工作台告警卡片通过 WebSocket 实时弹窗，伴随声音提示。分析师点击截图，确认"人员入侵"，点击"确认告警"并备注"已驱离"。
3. **结果**：告警状态变更为"已处理"，处置耗时记录入库，后续报表可追溯。

*场景 C：系统管理员 - 误报率治理*
1. **背景**：收到反馈，门口树木晃动频繁触发"区域越界"告警。
2. **操作**：管理员进入"检测项目"，在预览画面上重新绘制 ROI 区域，避开树木，并将灵敏度从 0.8 下调至 0.6。
3. **结果**：配置实时生效，次日同类误报减少 95%。

## 4. 功能需求详解

### 4.1 工作台模块
**功能描述**：作为用户登录后的首页，提供全局态势感知与高频操作入口，采用深色主题减少视觉疲劳。

| 功能点 | 详细描述 | 前置条件 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **核心指标卡片** | 展示：在线设备数、今日告警总数、待处理告警数、平均响应时长。数据每 10s 自动刷新。 | 用户已登录 | 数字准确，趋势箭头 (↑↓) 颜色区分正负向。 |
| **实时告警流** | 顶部横幅或侧边栏，以 WebSocket 推送最新 5 条告警，点击可跳转详情。 | 存在活跃告警 | 新告警出现时有高亮/动画提示，无延迟。 |
| **快捷入口** | 提供"新增推流"、"创建检测"、"告警列表"三个核心按钮。 | 无 | 点击后 1s 内跳转至对应功能页。 |

### 4.2 推流管理模块
**功能描述**：统一管理视频流生命周期，支持多协议接入与状态监控。

#### 4.2.1 视频流接入
*   **支持协议**：RTSP, RTMP, HTTP-FLV。
*   **配置参数**：
    *   流名称（必填，唯一标识）
    *   推流地址（必填，格式校验）
    *   分辨率（1080P/720P/480P，默认 1080P）
    *   帧率（15/25/30 FPS，默认 25）
    *   编码格式（H.264/H.265，默认 H.264）
*   **操作流程**：点击"新增推流" -> 填写表单 -> 点击"探测" -> 系统返回流信息（分辨率/码率） -> 确认保存。
*   **预期结果**：保存成功后，列表新增一条记录，状态初始为"检测中"，3s 内变为"在线"或"离线"。

#### 4.2.2 流状态监控与批量操作
*   **状态定义**：在线（绿）、离线（灰）、异常（红，如断流/解码失败）。
*   **批量操作**：支持勾选多条流进行"批量启用/停用/删除"。
*   **预期结果**：批量操作需二次确认，执行进度条显示，完成后列表自动刷新。

### 4.3 检测项目管理
**功能描述**：将视频流与 AI 算法绑定，配置检测逻辑。

#### 4.3.1 算法与区域配置
*   **算法类型**：人员入侵、物品遗留、烟火检测、区域越界。
*   **ROI 绘制**：
    *   支持多边形、矩形框选。
    *   支持"排除区域"（如：在入侵检测中排除人行道）。
    *   支持坐标导入/导出（JSON）。
*   **灵敏度阈值**：滑块调节 0.1 - 0.9，步长 0.05。

#### 4.3.2 任务控制
*   **启停控制**：单任务开关，支持定时启停（MVP 暂不支持，V2.0 规划）。
*   **预期结果**：开启后，AI 推理服务在 5s 内加载配置；ROI 变更实时生效，无需重启流。

### 4.4 告警中心模块（核心）
**功能描述**：基于 WebSocket 的实时告警接收、展示、处置与统计闭环。

#### 4.4.1 实时告警推送
*   **机制**：后端 AI 命中事件 -> 消息队列 -> WebSocket 网关 -> 前端。
*   **内容**：告警 ID、时间戳、流名称、算法类型、置信度、截图 URL、ROI 坐标。
*   **交互**：
    *   列表顶部插入新告警，伴随淡入动画。
    *   浏览器标签页 Title 闪烁提示。
    *   可选：浏览器通知（需授权）。

#### 4.4.2 告警列表与筛选
*   **展示字段**：时间、位置、类型、截图缩略图、置信度、状态（待处理/已确认/已忽略/误报）。
*   **筛选条件**：时间范围、流名称、算法类型、置信度区间、处理状态。
*   **搜索**：支持告警 ID 模糊搜索。

#### 4.4.3 告警处置流程
*   **操作**：
    1.  **确认**：标记为真实事件，可填写处置备注。
    2.  **忽略**：标记为低价值事件，不纳入统计。
    3.  **误报**：标记为算法错误，触发"负样本反馈"（V2.0 自动回传训练）。
*   **预期结果**：操作后状态立即变更，列表排序更新（待处理置顶）。

#### 4.4.4 告警统计
*   **维度**：按小时/天/周、按算法类型、按流名称。
*   **图表**：折线图（趋势）、饼图（类型分布）、柱状图（Top 5 告警流）。

### 4.5 系统管理
*   **用户权限**：RBAC 模型，预设"管理员"、"运维"、"分析师"角色，支持自定义权限点。
*   **操作日志**：记录所有增删改操作，包含：操作人、IP、时间、模块、动作、结果。保留 90 天。
*   **系统配置**：WebSocket 心跳间隔、告警截图保留天数、全局灵敏度基准。

## 5. 工单分发管理

**功能描述**：将 AI 告警事件转化为可追踪、可流转的工单，支持自动分发、人工指派、处理反馈与全生命周期跟踪，形成"告警 → 工单 → 处置 → 闭环"的完整链路。

### 5.1 工单自动生成

| 功能点 | 详细描述 | 前置条件 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **告警转工单规则** | 支持配置自动转工单条件：按算法类型、置信度阈值（≥ 80%）、流名称、连续命中次数等维度触发。 | 告警中心已产生符合规则的告警 | 系统自动创建工单，工单编号格式 `WO-YYYYMMDD-XXXXX`，5s 内完成创建。 |
| **工单内容** | 包含：工单编号、关联告警 ID、算法类型、截图、发生时间、推流名称、ROI 区域坐标、置信度、自动生成的优先级（高/中/低）。 | 自动转工单规则匹配 | 工单详情页完整展示上述字段，截图可点击放大。 |
| **手动创建** | 支持用户在告警列表或工单列表中手动勾选多条告警，一键"合并创建工单"。 | 至少勾选 1 条告警 | 合并工单标题自动拼接，内容聚合多条告警信息，支持人工编辑标题和描述。 |

### 5.2 工单分配与流转

| 功能点 | 详细描述 | 前置条件 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **自动分配策略** | 支持三种分配模式：①按人员忙闲度（最少待处理工单优先）；②按技能组轮询（RR 轮询）；③按值班表（预定义排班）。 | 工单已创建且状态为"待分配" | 系统自动匹配责任人，工单状态变更为"处理中"，责任人收到 WebSocket 通知及（可选）企微/钉钉消息。 |
| **手动指派** | 分配人或管理员可将工单指派给任意在线人员，支持多人协办（主责 + 协办）。 | 工单状态为"待分配"或"处理中" | 被指派人收到通知，工单详情显示主责人及协办人列表。 |
| **转派** | 处理人因故无法处理时，可申请转派，填写转派原因后提交管理员审批或自动转派至同组其他成员。 | 工单状态为"处理中" | 原处理人失去工单操作权限，新处理人接手，操作日志记录转派轨迹。 |
| **工单流转状态** | 待分配 → 处理中 → 待验收 → 已关闭 / 已驳回。 | — | 状态变更自动触发通知，超时未处理（可配置阈值）自动升级告警通知上级。 |

### 5.3 工单处置与反馈

| 功能点 | 详细描述 | 前置条件 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **处置操作** | 处理人可上传现场照片、填写处置描述、标记处置结果（已排除/误报/需跟进）。 | 工单状态为"处理中" | 处置记录追加至工单时间线，支持多轮补充。 |
| **验收闭环** | 处理完成后提交"待验收"，由分配人或管理员审核，可选择"通过"或"驳回"（驳回需填写原因）。 | 工单状态为"待验收" | 通过 → 状态变更为"已关闭"；驳回 → 回退至"处理中"，原处理人重新处置。 |
| **SLA 监控** | 按优先级设定处理时限（高：30min，中：2h，低：8h），超时后工单标记"超时"，自动升级通知上级。 | 工单已分配 | 工单列表"超时"列红色高亮，仪表盘展示 SLA 达标率。 |
| **知识库关联** | 处置完成后，支持将处置方案一键保存至知识库，供同类工单参考。 | 工单状态为"已关闭" | 知识库条目自动打标签（算法类型、位置），支持全文搜索。 |

### 5.4 工单列表与看板

| 功能点 | 详细描述 | 前置条件 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **工单列表** | 展示字段：编号、标题、优先级、状态、责任人、创建时间、SLA 剩余时间。支持多条件筛选（状态、优先级、时间范围、算法类型）。 | 用户已登录 | 列表支持分页（默认 20 条/页），支持按创建时间、优先级、状态排序。 |
| **工单看板（Kanban）** | 以看板视图按状态分列（待分配 / 处理中 / 待验收 / 已关闭），支持拖拽变更状态。 | 用户已登录 | 拖拽后自动更新工单状态并触发通知，支持看板与列表视图一键切换。 |
| **工单统计报表** | 按日/周/月维度统计：工单总数、完成率、平均处理时长、SLA 达标率、Top 5 高频问题类型。 | 存在已关闭工单 | 图表以折线图 + 柱状图展示，支持导出 CSV/Excel。 |

## 7. 非功能需求

### 7.1 性能要求

| 指标 | 要求 | 备注 |
| :--- | :--- | :--- |
| **并发路数** | ≥ 50 路 1080P@25FPS | 单节点，GPU 资源充足前提下 |
| **端到端延迟** | ≤ 3 秒 | 从摄像头采集到前端弹窗 |
| **告警推送延迟** | ≤ 500ms | 后端生成到前端接收 |
| **页面加载** | ≤ 2 秒 | 首屏 FCP |
| **API 响应** | P99 ≤ 300ms | 列表查询、配置保存 |

### 7.2 安全性要求
1.  **传输加密**：全站 HTTPS，WebSocket 使用 WSS。
2.  **鉴权**：JWT Token，有效期 2h，支持 Refresh Token。
3.  **数据脱敏**：日志中禁止明文记录推流地址密码。
4.  **接口防护**：关键写接口需 CSRF Token，限流 100 QPS/IP。

### 7.3 可靠性要求
1.  **流媒体服务**：支持断流自动重连（重试 3 次，间隔 5s）。
2.  **AI 服务**：单路推理崩溃不影响其他路，自动重启容器。
3.  **消息队列**：WebSocket 断连期间，告警消息不丢失，重连后补推最近 10 条。

### 7.4 兼容性要求
*   **浏览器**：Chrome 90+，Edge 90+，Firefox 88+（不支持 IE）。
*   **分辨率**：适配 1920x1080 及以上，支持 125%/150% 系统缩放。

## 8. 数据需求

### 8.1 数据存储方案
*   **关系型数据**：PostgreSQL（用户、组织架构、角色权限、配置、告警元数据、工单、排班）。
*   **时序/日志数据**：ClickHouse 或 Elasticsearch（告警统计、操作日志、工单 SLA 记录）。
*   **对象存储**：MinIO / S3（告警截图、工单处置附件、短视频片段）。
*   **缓存**：Redis（WebSocket 会话、实时指标、Token、在线状态、忙闲度）。

### 8.2 数据保留策略
*   **告警截图**：默认 30 天，可配置。
*   **告警记录**：永久保留，冷数据（>1 年）归档。
*   **操作日志**：90 天在线，之后归档。

### 8.3 数据导出功能
*   **格式**：CSV, Excel, PDF（报表）。
*   **限制**：单次导出上限 10,000 条，异步生成，下载中心通知。

## 11. 风险与约束

### 11.1 技术风险
1.  **GPU 资源瓶颈**：50 路并发对显存要求高，需做动态 Batch 调度，防止 OOM。
    *   **对策**：引入 TensorRT 加速，设置显存水位线，超限时自动降级低优先级流。
2.  **WebSocket 连接风暴**：大量用户同时在线可能导致网关压力。
    *   **对策**：采用 Redis Pub/Sub 多节点广播，前端实现断线指数退避重连。
3.  **流协议兼容性**：部分老旧摄像头 RTSP 信令不规范。
    *   **对策**：集成 FFmpeg 兼容层，提供"流诊断"工具，输出错误码指引。

### 11.2 业务风险
1.  **误报率过高**：导致用户关闭告警通知，产品价值归零。
    *   **对策**：MVP 阶段默认开启"二次确认"逻辑（连续 3 帧命中才报警），提供"误报反馈"快捷按钮。
2.  **隐私合规**：人脸/车牌数据未脱敏。
    *   **对策**：MVP 默认开启人脸/车牌模糊滤镜，管理员可手动关闭（需审计日志）。

### 11.3 资源约束
1.  **MVP 周期**：8 周开发，4 周测试，必须严格砍需求。
2.  **硬件成本**：客户需自备服务器资源，初期最低 1 台 GPU 服务器（如 NVIDIA A10/RTX 系列）+ 应用服务器 + 数据库服务器。平台提供部署安装包与实施指导。
